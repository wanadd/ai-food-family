# Roadmap v1: ПланАм

План развития продукта по этапам. Каждый этап завершается **проверяемым результатом для пользователя**, а не только техническим PR.

**Легенда статуса** (ориентир на текущую кодовую базу):

- ✅ — базовый функционал уже есть
- 🟡 — частично реализовано, требуется доводка по видению
- ⬜ — запланировано

---

## Этап 1. Профиль и настройки

**Цель:** единая точка «кто я в ПланАм» и как приложение со мной общается.

| Задача | Статус |
|--------|--------|
| Вход через Telegram + подтверждение телефона в боте | ✅ |
| Страница профиля, имя, контакты | 🟡 |
| Переключение **Личный / Семейный** режим на главной | ✅ |
| Настройки уведомлений (покупки, готовка, время) | 🟡 |
| Язык, часовой пояс, единицы измерения | ⬜ |
| Политика приватности и удаление данных (понятным языком) | ⬜ |

**Результат этапа:** пользователь настраивает аккаунт за 2–3 минуты без технических терминов.

---

## Этап 2. Профиль питания

**Цель:** у каждого человека — свой профиль, на котором строятся меню и нутрициолог.

| Задача | Статус |
|--------|--------|
| Онбординг (цели, диета, аллергии, бюджет, время) | ✅ |
| Редактирование профиля после онбординга | 🟡 |
| Отдельный экран «Профиль питания» вне онбординга | ⬜ |
| Версионирование / история изменений ограничений | ⬜ |
| Связь профиля с виртуальным участником (без Telegram) | ⬜ |

**Результат этапа:** любое изменение «не ем глютен» сразу отражается в меню и рекомендациях.

---

## Этап 3. Семья и участники

**Цель:** опциональная семья, пара или группа; реальные и виртуальные участники.

| Задача | Статус |
|--------|--------|
| Создание семьи, роли | ✅ |
| Приглашения: телефон, контакт в боте, deep-link | ✅ |
| Виртуальные участники (имя, возраст, ограничения) | 🟡 |
| Группа без «семейной» терминологии (соседи, друзья) | ⬜ |
| Управление составом: выход, смена роли, архив | 🟡 |

**Результат этапа:** можно жить в личном режиме или собрать контекст на 2–10 человек без обязательной регистрации всех в Telegram.

---

## Этап 4. Главная вкладка ПланАм

**Цель:** один экран — «что сегодня важно».

| Задача | Статус |
|--------|--------|
| Приветствие, режим, быстрые действия | ✅ |
| Карточка выбранного меню / ссылка на план | ✅ |
| Сводка покупок (сколько позиций осталось) | ✅ |
| Блок «заканчивается в запасах» | ⬜ |
| Мини-виджет прогресса цели (нутрициолог) | ⬜ |
| Нижняя навигация (главная, покупки, запасы, меню, профиль) | ✅ |

**Результат этапа:** с главной решается 80% ежедневных задач в 1–2 тапа.

---

## Этап 5. Меню

**Цель:** AI-меню с учётом всех профилей, запасов, остатков и числа персон.

| Задача | Статус |
|--------|--------|
| Генерация вариантов меню (AI) | ✅ |
| Выбор и сохранение меню на период | ✅ |
| Учёт целей, аллергий, ограничений в промпте | ✅ |
| Учёт запасов и остатков блюд при генерации | 🟡 |
| Порции по числу персон (вкл. виртуальных) | 🟡 |
| Календарь / план по дням с заменой блюда | ⬜ |
| Экспорт «что готовить сегодня» в уведомление | 🟡 |

**Результат этапа:** меню практичное — готовить из того, что есть, докупать только недостающее.

---

## Этап 6. Покупки

**Цель:** универсальный список покупок с автопереносом продуктов в запасы.

| Задача | Статус |
|--------|--------|
| Категории, поиск, CRUD позиций | ✅ |
| Пользовательские категории, флаг «еда / не еда» | ✅ |
| Отметка «куплено» → автозапасы (продукты) | ✅ |
| Не продуктовые товары без автозапасов | ✅ |
| Генерация списка из меню | 🟡 |
| Совместное редактирование в семейном режиме | 🟡 |
| Шаблоны списков («еженедельный», «к празднику») | ⬜ |

**Результат этапа:** один список на дом — от молока до стирального порошка; продукты сами попадают в запасы.

---

## Этап 7. Запасы

**Цель:** актуальная картина «что есть дома».

| Задача | Статус |
|--------|--------|
| Список по категориям, фильтры | ✅ |
| Источники: покупки, вручную, чек, голос | 🟡 |
| Merge по названию + единица + scope | ✅ |
| Срок годности / «скоро испортится» | ⬜ |
| Остатки приготовленных блюд | ⬜ |
| Влияние на меню (явная подсказка в UI) | 🟡 |

**Результат этапа:** пользователь видит запасы и понимает, как они меняют меню и покупки.

---

## Этап 8. Нутрициолог

**Цель:** панель рекомендаций, целей, прогресса и заботы — не «ещё один чат».

| Задача | Статус |
|--------|--------|
| Цели из онбординга отображаются в UI | 🟡 |
| Панель рекомендаций (карточки, не поток сообщений) | ⬜ |
| Прогресс к цели (вес, привычки, streak) | ⬜ |
| Мягкие советы с учётом меню и запасов | ⬜ |
| Опциональный чат с AI внутри раздела | ⬜ |

**Результат этапа:** раздел отвечает на «как я двигаюсь к цели» без ручного дневника.

---

## Этап 9. AI Care System

**Цель:** умные напоминания и вмешательства без спама.

| Задача | Статус |
|--------|--------|
| Напоминания: купить, готовить, время | ✅ |
| Контекстные триггеры (запасы, меню, покупки) | 🟡 |
| Правила частоты и «тихие часы» | ⬜ |
| Семейные дайджесты (одно сообщение на дом) | ⬜ |
| Обратная связь: «полезно / не сейчас» | ⬜ |

**Результат этапа:** уведомления ощущаются как забота, а не как навязчивый бот.

---

## Этап 10. Подписки и Амы

**Цель:** монетизация и прозрачный расход AI-ресурсов.

| Тариф | Задачи |
|-------|--------|
| **Личный** | ⬜ Оформление, лимиты, продление |
| **Совместный** | ⬜ Два профиля, общий контекст |
| **Семейный** | ⬜ Семья + виртуальные участники |
| **ПланАм PRO** | см. этап 11 |

| Задача | Статус |
|--------|--------|
| Баланс **Ам**, история списаний | ⬜ |
| Стоимость действий в человеческих формулировках | ⬜ |
| Пополнение Ам / пакеты в подписке | ⬜ |
| Платёж через Telegram / внешний провайдер | ⬜ |

**Результат этапа:** пользователь понимает, за что платит и сколько AI-действий осталось.

---

## Этап 11. PRO-режим

**Цель:** углублённый контроль для спорта, здоровья, веса и семейной аналитики.

| Задача | Статус |
|--------|--------|
| Расширенные метрики (вес, активность, КБЖУ) | ⬜ |
| Связка с целями нутрициолога | ⬜ |
| Семейная сводка прогресса | ⬜ |
| Аналитика: тренды, отчёты за период | ⬜ |
| Экспорт для врача / тренера (PDF, ссылка) | ⬜ |

**Результат этапа:** PRO даёт измеримую пользу поверх базового цикла «меню → покупки → запасы».

---

## Этап 12. Админ-панель

**Цель:** операционное управление продуктом без доступа к БД.

| Задача | Статус |
|--------|--------|
| Пользователи, семьи, блокировки | ⬜ |
| Подписки, Амы, ручные начисления | ⬜ |
| Модерация контента (рецепты, категории) | ⬜ |
| Логи AI-запросов и ошибок (без PII в UI) | ⬜ |
| Feature flags для поэтапного rollout | ⬜ |

**Результат этапа:** поддержка и продукт решают инциденты без миграций «руками».

---

## Этап 13. Резервное копирование и откат

**Цель:** сохранность данных пользователей и быстрое восстановление после сбоев.

| Задача | Статус |
|--------|--------|
| Автоматический бэкап PostgreSQL (расписание, хранение) | 🟡 |
| Проверка восстановления (drill) | ⬜ |
| Бэкап Redis / критичных конфигов | ⬜ |
| Point-in-time recovery (по возможности хостинга) | ⬜ |
| Откат релиза приложения (образы, миграции) | 🟡 |
| Экспорт данных пользователем («мои данные») | ⬜ |

**Результат этапа:** документированный RPO/RTO; восстановление проверено, а не только настроено.

---

## Рекомендуемый порядок работ

```text
1–3  Профиль, питание, участники     ← фундамент контекста
4    Главная                          ← ежедневная точка входа
5–7  Меню → Покупки → Запасы          ← основной цикл продукта
8–9  Нутрициолог + AI Care            ← удержание и цели
10–11 Подписки, Амы, PRO              ← монетизация
12–13 Админка, бэкапы                 ← эксплуатация
```

Параллельно на всех этапах:

- доводка **Telegram-бота** (OCR чеков, голос, приглашения);
- соблюдение принципа **лёгкого UI** из [PRODUCT_VISION.md](./PRODUCT_VISION.md).

---

## Критерии готовности v1

Версия **v1** считается готовой, когда:

1. Личный режим проходит полный цикл без семьи.
2. Семейный режим с реальными и виртуальными участниками влияет на меню и покупки.
3. Покупки → запасы работают автоматически для продуктов.
4. Нутрициолог и AI Care дают измеримую пользу, а не только чат.
5. Подписки и Амы прозрачны для пользователя.
6. PRO открывает расширенную аналитику без поломки базового сценария.
7. Админка и бэкапы позволяют безопасно эксплуатировать продукт в production.
